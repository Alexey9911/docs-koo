Pixie simulates all transactions before execution to prevent failures and unexpected outcomes.

## How Simulation Works

Before presenting a transaction for approval, Pixie runs a pre-flight simulation on Solana's RPC:

```typescript
const simulation = await connection.simulateTransaction(transaction, {
  commitment: 'confirmed',
  sigVerify: false
});

if (simulation.value.err) {
  throw new Error(`Simulation failed: ${simulation.value.err}`);
}
```

## What Gets Validated

| Check | Description |
|-------|-------------|
| **Sufficient Balance** | Wallet has enough tokens for transaction |
| **Slippage Tolerance** | Output meets minimum acceptable amount |
| **Compute Budget** | Transaction fits within compute limits |
| **Account Existence** | Required accounts are initialized |
| **Program Errors** | No smart contract failures |

## Error Detection

Common errors caught by simulation:

| Error | Cause | Solution |
|-------|-------|----------|
| **Insufficient Funds** | Not enough tokens | Add funds or reduce amount |
| **Slippage Exceeded** | Price moved too much | Increase slippage or retry |
| **Account Not Found** | Missing token account | Create account (auto-handled) |
| **Program Error** | Smart contract failure | Contact support |

<Warning>
  If simulation fails, the transaction will not be presented for approval. This prevents wasted transaction fees on failed operations.
</Warning>

## Compute Unit Estimation

Simulation provides accurate compute unit usage:

```typescript
const computeUnits = simulation.value.unitsConsumed;

// Add 20% buffer for safety
const computeLimit = Math.ceil(computeUnits * 1.2);
```

**Typical Compute Units:**
- Simple transfer: ~5,000 units
- Token swap: ~150,000 units
- Complex DeFi: ~200,000 units

## Performance

| Metric | Value |
|--------|-------|
| Simulation Time | ~12ms |
| Success Rate | 99.7% |
| False Positives | <0.1% |

<Note>
  Simulation uses current blockchain state. Rapid price movements between simulation and execution may cause actual results to differ slightly.
</Note>

## Limitations

Simulation cannot predict:

- Future price movements
- Network congestion changes
- MEV attacks (front-running)
- Validator behavior

## Best Practices

1. **Review simulation results** before approving transactions
2. **Increase slippage** if simulation shows high price impact
3. **Retry failed simulations** after a few seconds
4. **Check compute units** for complex transactions

## Simulation vs Execution

| Aspect | Simulation | Execution |
|--------|------------|-----------|
| **Cost** | Free | ~0.000005 SOL |
| **State Changes** | None | Permanent |
| **Timing** | Instant | ~400ms |
| **Accuracy** | 99.7% | 100% |
